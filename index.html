Phase 01 · Weeks 1–3
OOP Foundations — The Right Way
Most engineers with 4+ years think they know OOP. They know the definitions. That's different from knowing how to use OOP to make design decisions. This phase fixes that gap.

Here's the honest truth about OOP at your level: you probably know what encapsulation is. But can you look at a class and immediately identify what it's doing wrong with encapsulation? Can you refactor it? That's the gap we're closing. In an LLD interview, you won't be asked "what is polymorphism?" You'll be asked to design a payment system, and you'll need to use polymorphism without being told to.

Week 1 is entirely Encapsulation + Abstraction. Week 2 is Inheritance + the misuse of it. Week 3 is Polymorphism + Composition vs Inheritance. Follow this order exactly — each one builds on the previous.

Week 1 · Days 1–7

Encapsulation & Abstraction

Start with encapsulation because it's the principle you'll violate most often without realising it. The sign of weak encapsulation is: public fields, setters on everything, and classes that let any caller change any state. When you see that, you should feel discomfort. That discomfort is your design instinct developing.

For abstraction: the key insight is that interfaces don't exist to make code "more complex" — they exist to let you swap implementations. If you can only ever have one implementation, you don't need an interface. If you'll have multiple, or if you want to test it, you need one. Every time you write an interface, ask yourself: will this ever have more than one implementation?

Day 1–2 Task: Encapsulation drill

  • Write a BankAccount class with broken encapsulation (public fields, no validation)
  • Write 3 tests that show how the broken version can be corrupted
  • Refactor it to have proper encapsulation — private fields, validated methods only
  • Write the same 3 tests — they should now fail to compile or pass safely
  • Write in your design journal: what did encapsulation protect here?
C# · Encapsulation — Bad vs Good Week 1 Reference
// ❌ BAD — Broken encapsulation
public class BankAccount {
    public decimal Balance;          // anyone can corrupt this
    public string OwnerName;          // no validation
    public bool IsActive;
}
// Caller can do: account.Balance = -99999; — disaster!

// ✅ GOOD — Properly encapsulated
public class BankAccount {
    private decimal _balance;
    private bool _isActive = true;

    public decimal Balance => _balance;     // read-only outside
    public string OwnerName { get; }       // set only via constructor

    public BankAccount(string owner, decimal initialDeposit) {
        if (string.IsNullOrWhiteSpace(owner)) throw new ArgumentException("Owner required");
        if (initialDeposit < 0) throw new ArgumentException("Deposit must be positive");
        OwnerName = owner;
        _balance = initialDeposit;
    }

    public void Deposit(decimal amount) {
        EnsureActive();
        if (amount <= 0) throw new ArgumentException("Must be positive");
        _balance += amount;
    }

    public void Withdraw(decimal amount) {
        EnsureActive();
        if (amount > _balance) throw new InvalidOperationException("Insufficient funds");
        _balance -= amount;
    }

    private void EnsureActive() {
        if (!_isActive) throw new InvalidOperationException("Account is closed");
    }
}
// Key: state can only change through validated methods. Invariants are protected.

Day 3–4 Task: Abstraction drill

  • Design an IPaymentProcessor interface — list only what callers need to know
  • Build 3 implementations: CreditCardProcessor, UpiProcessor, NetBankingProcessor
  • Write a CheckoutService that depends ONLY on IPaymentProcessor — never the concrete types
  • Write a unit test that uses a mock IPaymentProcessor — this tests that your abstraction is correct
  • Ask yourself: if I add a 4th payment method, does anything in CheckoutService change? Answer should be NO
Week 2 · Days 8–14

Inheritance — Use & Misuse

Inheritance is the most misused OOP concept. Most code I review has inheritance chains that should be composition instead. The rule is simple: only inherit when you genuinely mean "B IS A type of A". Not "B uses A". Not "B is sort of like A". IS A.

A Dog IS AN Animal — correct. A UserRepository IS NOT A Database — that's using a database, not being one. If you catch yourself inheriting to reuse code rather than to model a genuine type relationship, stop. Use composition.

This week you'll intentionally write bad inheritance, then fix it. That contrast is where the learning lives.

Day 8–10 Task: Inheritance done right

  • Create an abstract Shape class with abstract Area() and Perimeter() methods
  • Implement Circle, Rectangle, Triangle — each overrides both methods
  • Write a method PrintShapeInfo(Shape s) that works with any shape — this demonstrates polymorphism via inheritance
  • Now add a ColoredShape — should it inherit Shape or contain one? Think carefully before coding
  • Answer: Contain one. ColoredShape HAS-A Shape. It's NOT A Shape that just happens to have color.
C# · Inheritance vs Composition — Critical Choice Week 2 Reference
// ✅ Correct inheritance — genuine is-a relationship
public abstract class Animal {
    public string Name { get; }
    protected Animal(string name) => Name = name;
    public abstract void MakeSound();
    public virtual void Move() => Console.WriteLine($"{Name} moves");
}
public class Dog : Animal {        // Dog IS AN Animal ✓
    public Dog(string name) : base(name) {}
    public override void MakeSound() => Console.WriteLine("Woof!");
}

// ❌ Wrong — inheriting to reuse code, not model hierarchy
public class Stack<T> : List<T> {   // Stack IS NOT A List!
    public void Push(T item) => Add(item);
    public T Pop() { var t = this[Count-1]; RemoveAt(Count-1); return t; }
    // Problem: exposes Insert, RemoveAt, etc — Stack shouldn't allow these!
}

// ✅ Fixed — composition. Stack USES a list, it's not a list
public class Stack<T> {
    private readonly List<T> _items = new();
    public void Push(T item) => _items.Add(item);
    public T Pop() {
        if (_items.Count == 0) throw new InvalidOperationException("Empty stack");
        var top = _items[^1];
        _items.RemoveAt(_items.Count - 1);
        return top;
    }
    public T Peek() => _items.Count > 0 ? _items[^1] : throw new InvalidOperationException();
    public int Count => _items.Count;
    // Clean interface. No leaked List methods.
}
Week 3 · Days 15–21

Polymorphism — The Heart of LLD

Polymorphism is what makes design patterns possible. Without it, every pattern collapses into a mess of if-else chains. The critical skill here isn't writing polymorphic code — you already do that. The skill is recognising when an if-else chain should be replaced with polymorphism. That recognition is what separates average engineers from strong LLD designers.

There's a rule I call the "type switch smell": if you ever see if (type == "X") ... else if (type == "Y") or switch(someEnum) in business logic, that's almost always a polymorphism opportunity. Your job this week is to find those in your own Ascendion code and fix them.

Day 15–17 Task: Replace conditionals with polymorphism

  • Write this broken code first: a TaxCalculator with a big switch statement on country code
  • Identify the polymorphism opportunity: each country's tax logic = a separate class
  • Refactor: ITaxStrategy interface, IndianTax, USTax, UKTax implementations
  • Now adding a new country means ZERO changes to existing code — only adding a new class
  • This pattern appears constantly in LLD interviews. Master this refactoring cold.

Day 18–21 Task: OOP capstone mini-project

  • Build a simple NotificationSystem from scratch — no googling
  • It must support Email, SMS, Push channels
  • Adding a new channel should require zero changes to existing classes
  • Each notification has content, recipient, priority level
  • Time yourself: first attempt may take 2–3 hours. That's fine.
  • You're ready for Phase 2 when: you can rebuild this from scratch in 30 minutes

Phase 1 exit criteria: Can you design a 3-class system using all four OOP pillars correctly, explain every design choice, and identify what's wrong with a poorly designed class — without referring to any notes? If yes, proceed to Phase 2.

Phase 02 · Weeks 4–5
SOLID — Not a Checklist, a Design Language
SOLID is the vocabulary senior engineers use to evaluate and critique designs. Learning to spot violations is more valuable than memorising definitions. This phase teaches you to read code and immediately see SOLID issues.

Here's how we're going to learn SOLID differently. For each principle, I'm going to show you a codebase that violates it. Your job is to first feel why it's bad — what problems does it cause? What breaks when requirements change? Then we fix it. The violation is the teacher, not the principle statement.

Spend 2 days on each principle: Day 1 is the violation + analysis. Day 2 is the fix + tests. Ten days total for 5 principles. Don't compress this — each principle is a distinct mental model you need to own.

Principle 1 — SRP

Single Responsibility Principle

"A class should have only one reason to change." — the most misunderstood line in software engineering. It doesn't mean one method. It means one actor — one stakeholder whose requirements drive change.

The practical question to ask about every class: "If I need to change this class, what caused the change?" If the answer is "we changed the email template" OR "we changed the database schema" OR "we changed the business logic" — that class has multiple reasons to change. That's an SRP violation.

In your .NET world, the most common SRP violation is the fat service class: it does validation, business logic, database calls, and email sending all in one place. Sound familiar? Think about your Ascendion codebase right now.

C# · SRP Violation → Fix Week 4 Reference
// ❌ VIOLATION — UserService has 4 reasons to change
public class UserService {
    private readonly SqlConnection _conn;

    public User GetUser(int id) {           // DB reason
        return _conn.Query<User>("SELECT * FROM Users WHERE Id=@id", new { id });
    }
    public bool ValidateUser(User u) {        // Validation reason
        return !string.IsNullOrEmpty(u.Email) && u.Age >= 18;
    }
    public void SendWelcomeEmail(User u) {   // Email reason
        var smtp = new SmtpClient();
        smtp.Send(u.Email, "Welcome!", "Hello " + u.Name);
    }
    public string FormatUserReport(User u) { // Formatting reason
        return $"User: {u.Name}, Email: {u.Email}, Age: {u.Age}";
    }
}

// ✅ FIXED — Each class has one reason to change
public class UserRepository {              // Only DB logic here
    private readonly IDbConnection _conn;
    public User GetUser(int id) => _conn.QueryFirst<User>("SELECT...", new { id });
    public void Save(User u) { /* ... */ }
}

public class UserValidator {               // Only validation here
    public bool IsValid(User u) =>
        !string.IsNullOrEmpty(u.Email) && u.Age >= 18;
}

public class UserEmailService {             // Only email here
    private readonly IEmailSender _sender;
    public void SendWelcome(User u) => _sender.Send(u.Email, "Welcome!", "...");
}

public class UserReportFormatter {          // Only formatting here
    public string Format(User u) => $"User: {u.Name}, ...";
}
// Now: change DB schema → touch only UserRepository. Change email template → touch only UserEmailService.

SRP Task (2 days):

  • Find a class in your Ascendion project that violates SRP. If you can't find one, it means you're not looking closely enough — it's always there
  • List every "reason to change" that class has. Write them down
  • Refactor it: each reason becomes a separate class
  • Write unit tests for each new class — notice how much easier testing is now
  • Journal: "What would have broken if I needed to change the DB logic before this refactor?"
Principle 2 — OCP

Open/Closed Principle

OCP is about making your code resilient to new requirements. The telltale violation: every new feature requires you to modify an existing class. That's dangerous because you're touching tested, working code. OCP says: design so new features are added by adding new code, not modifying existing code.

The pattern that makes OCP work is almost always: replace the thing that varies (the if-else logic) with an interface, then put each variant in its own class. This is the Strategy pattern at its core — and it's why OCP and Strategy are inseparable.

C# · OCP — Adding Features Without Modification Week 4 Reference
// ❌ VIOLATION — Every new report type requires modifying this class
public class ReportGenerator {
    public string Generate(Report data, string format) {
        if (format == "PDF") { return GeneratePdf(data); }
        else if (format == "Excel") { return GenerateExcel(data); }
        else if (format == "CSV") { return GenerateCsv(data); }
        // New format = modify this method = risk breaking existing formats!
        throw new NotSupportedException(format);
    }
}

// ✅ FIXED — New format = new class only, zero existing changes
public interface IReportFormatter {
    string Format { get; }
    string Generate(Report data);
}

public class PdfReportFormatter : IReportFormatter {
    public string Format => "PDF";
    public string Generate(Report data) { return "...pdf bytes..."; }
}
public class ExcelReportFormatter : IReportFormatter {
    public string Format => "Excel";
    public string Generate(Report data) { return "...xlsx bytes..."; }
}

public class ReportGenerator {
    private readonly IReadOnlyDictionary<string, IReportFormatter> _formatters;

    public ReportGenerator(IEnumerable<IReportFormatter> formatters)
        => _formatters = formatters.ToDictionary(f => f.Format);

    public string Generate(Report data, string format) {
        if (!_formatters.TryGetValue(format, out var formatter))
            throw new NotSupportedException(format);
        return formatter.Generate(data);
    }
}
// Adding Word format? Just add WordReportFormatter. ReportGenerator never changes.

OCP Task: Write a discount engine for an e-commerce system. Start with violations (giant switch on customer type), then refactor to be OCP-compliant. Then add a new "Seasonal" discount type — if you have to modify more than 1 file, your OCP isn't solid yet.

Principle 3 — LSP

Liskov Substitution Principle

LSP is the principle that makes inheritance safe. The test: take any code that uses a base type. Now replace it with a derived type without telling the code. Does it still work correctly? If it fails or behaves unexpectedly — you've violated LSP.

The Square/Rectangle classic violation: Rectangle has Width and Height set independently. Square forces Width == Height. Any code that sets them independently will produce wrong results with a Square. Square cannot substitute Rectangle safely. Fix: don't inherit. Both implement an IShape interface instead.

Red flag in code review: a derived class throws NotImplementedException for a method inherited from the base. That's LSP violated loudly.

LSP Task: Build the Square/Rectangle violation in C#. Write a test: SetDimensions(Rectangle r) { r.Width = 4; r.Height = 5; Assert.Equal(20, r.Area()); }. Pass in a Square. Watch it fail. Now fix the hierarchy so both implement IShape instead. Re-run the test — the fix should force you to change the test too (which is fine — that's the correct design).

Principles 4 & 5 — ISP & DIP

Interface Segregation + Dependency Inversion

ISP: If an implementing class has methods it doesn't use, your interface is too fat. The test: can you implement this interface without any throw new NotImplementedException()? If not — split the interface. In C# with DI, narrow interfaces are better for testability too: a class that needs only IReader shouldn't have to take an IRepository just because it lives there.

DIP: This is what makes your code testable and your architecture clean. The rule: high-level modules (business logic) should never import from low-level modules (database, email, filesystem). Both should depend on abstractions. In ASP.NET Core, this is literally how the DI container is designed. You're already doing this — now understand WHY. Every time you register a service as an interface in Program.cs, you're applying DIP.

ISP + DIP Combined Task:

  • Take a fat IWorkerRepository interface that has Read, Write, Delete, Archive, Export methods
  • A read-only analytics service only needs Read — it's forced to take the full interface
  • Segregate: IReadableRepository, IWritableRepository, IDeletable, IArchivable, IExportable
  • The analytics service now takes only IReadableRepository — DIP applied cleanly
  • Write a unit test for the analytics service using a mock IReadableRepository — notice how easy it is now

Phase 2 exit criteria: Given any class, can you identify which SOLID principles it violates and explain exactly why? Can you refactor any SOLID violation cold — without referring to notes? Can you write a unit test that becomes easier to write after a SOLID refactor?

Phase 03 · Weeks 6–10
Design Patterns — Learn the Problem, Not the Pattern
23 patterns. 5 weeks. But here's the crucial thing: patterns are solutions to recurring problems. If you learn the pattern without understanding the problem it solves, you'll never know when to apply it. For each pattern, I'll show you the problem first, then the pattern as the solution.

Most people memorise patterns in the abstract: "Singleton ensures one instance". That's useless. What you need to know is: "I'm creating a Logger in 5 different classes and getting 5 different instances — each one opens its own file handle. That's the problem. Singleton is the solution." Learn the smell that indicates a pattern, not just the pattern structure.

Schedule: 2 patterns per week, Monday/Wednesday. Thursday/Saturday: implement each with a realistic .NET scenario. Week 10: review all patterns — can you name the problem that requires each one? That's your exit test.

For each pattern, your practice cycle is: (1) Read the "signal" — what problem does it solve? (2) Write the violation code — code that needs the pattern but doesn't use it. (3) Implement the pattern to fix it. (4) Write a unit test that proves the pattern works. (5) Ask: "Where in my Ascendion codebase could this pattern have helped?"

Don't do all patterns in sequence like a textbook. Do each one until it's automatic, then move on.

C# · State Pattern — Order State Machine Week 8 Deep Dive
// The State pattern replaces this kind of mess:
// if (order.Status == "Pending") { /* 20 lines */ }
// else if (order.Status == "Processing") { /* 30 lines */ }
// — adding a new status means modifying this method. Brittle.

public interface IOrderState {
    void Confirm(Order order);
    void Ship(Order order);
    void Cancel(Order order);
    string StatusName { get; }
}

public class PendingState : IOrderState {
    public string StatusName => "Pending";
    public void Confirm(Order o) => o.TransitionTo(new ProcessingState());
    public void Ship(Order o) => throw new InvalidOperationException("Confirm first!");
    public void Cancel(Order o) => o.TransitionTo(new CancelledState());
}

public class ProcessingState : IOrderState {
    public string StatusName => "Processing";
    public void Confirm(Order o) => throw new InvalidOperationException("Already confirmed");
    public void Ship(Order o) => o.TransitionTo(new ShippedState());
    public void Cancel(Order o) => o.TransitionTo(new CancelledState());
}

public class Order {
    private IOrderState _state = new PendingState();
    public string Status => _state.StatusName;

    internal void TransitionTo(IOrderState newState) => _state = newState;
    public void Confirm() => _state.Confirm(this);
    public void Ship() => _state.Ship(this);
    public void Cancel() => _state.Cancel(this);
}
// Adding "RefundRequested" state = new class only. Zero changes to Order.

Phase 04 · Weeks 11–14
LLD Problems — The 5-Step Framework
Every LLD problem you'll ever face — in an interview or at work — can be solved with the same 5-step process. Internalise this process until it's automatic. It's not just a framework; it's a habit of mind.

Here's the brutal truth about LLD practice: most engineers approach problems the wrong way. They immediately start thinking about code — "OK so I need a ParkingLot class with an array of spots…" Wrong order. You're building before you understand. The result is a design that works for the happy path and breaks on every edge case.

The correct order is: Understand → Model → Design → Validate → Implement. The first two steps have nothing to do with code. They're about thinking. And they're what interviewers care about most — because a senior engineer who thinks before coding is rare and valuable.

This is your non-negotiable rule for every problem you practice: you are not allowed to write a single class or method until you've completed Steps 1 and 2 on paper or whiteboard.

1
Clarify Requirements — Ask 5 Questions Before Touching Code
For any problem, ask: (1) Who are the actors? (Who uses this system?) (2) What are the core actions each actor performs? (3) What are the scale constraints? (single machine? distributed?) (4) What's explicitly OUT of scope? (5) What are the key edge cases? — Write these down. In an interview, saying "let me ask a few questions first" signals senior thinking immediately. An engineer who jumps straight to coding is signalling junior thinking.
3–5 minutes in interview · Write before coding in practice
2
Identify Entities — Every Noun Is a Candidate Class
Read the requirements. Highlight every noun. Each noun is a candidate class. Then apply two filters: Does it have both state AND behaviour? (If only state, it might be a data object. If only behaviour, it might be a service.) Is it distinct enough to warrant its own class, or is it just an attribute of another class? A "Parking Spot Location" is probably just an attribute of ParkingSpot, not its own class. "Vehicle" has state (plate, type) and behaviour (enter, exit) — it's a class. Write your entity list before touching an IDE.
5 minutes · Always on paper/whiteboard first
3
Define Relationships — is-a vs has-a vs uses
For each pair of entities, ask: Is it inheritance (is-a)? Composition (has-a, lifecycle coupled)? Aggregation (has-a, lifecycle independent)? Association (uses, no ownership)? Draw this out. A ParkingLot HAS floors (composition — floors don't exist without the lot). A Floor HAS spots (composition). A Spot IS-A ParkingEntity of a specific size. A Vehicle USES a Spot (association — vehicle exists without the spot). This entity-relationship map becomes your class diagram.
5 minutes · Sketch on paper
4
Apply Patterns — Name Them and Justify Them
Look at your design. Where does behaviour vary? (Strategy). Where do you have state transitions? (State). Where do objects need to notify others? (Observer). Where are you creating families of objects? (Factory). Don't force patterns. One or two well-justified patterns per problem is ideal. Three is solid. Five is suspiciously forced. The key word is "justify" — in an interview, say "I'd use Strategy here because the fee calculation logic varies by vehicle type and we'll likely add more types" — not just "I'll use Strategy here".
5 minutes · Name patterns out loud in practice
5
Implement — Interfaces First, Then Key Classes
Start with interfaces — not classes. Define the contracts first. Then implement 3–4 key classes that show your design thinking most clearly. You don't need to implement everything — interviewers know. What they're evaluating is whether your code matches your design intent. Common mistake: people write getters/setters for 20 minutes and never get to the interesting logic. Skip boilerplate. Focus on the parts that show OOP thinking — polymorphic hierarchies, pattern implementations, state transitions.
20–30 minutes · Interfaces before classes always

Why Start Here

Step 1 — Your Clarifying Questions

Before reading ahead, pause and ask yourself: What types of vehicles? Multiple floors? Hourly vs flat pricing? Handicapped spots? Entry/exit gates? Reserved spots? This mental habit of questioning is what you practice with every problem.

Step 2 — Entity List
  • ParkingLot — the top-level container. Has floors, entry/exit gates
  • ParkingFloor — has spots. Knows how many are available
  • ParkingSpot — abstract. Has ID, spot type, availability status
  • CompactSpot, LargeSpot, HandicappedSpot — concrete spots (inherit ParkingSpot)

  • Car, Truck, Motorcycle — concrete vehicles
  • ParkingTicket — issued on entry. Has entry time, spot, vehicle

  • ParkingLot COMPOSES ParkingFloors (floors don't exist without lot)
  • ParkingFloor COMPOSES ParkingSpots

  • CompactSpot INHERITS ParkingSpot (is-a)
  • ParkingTicket ASSOCIATES with Vehicle and ParkingSpot (uses, doesn't own)

Chess Game
Intermediate FAANG Favourite
Why Chess Is the Ultimate LLD Test

Chess requires all of polymorphism (each piece moves differently), Command pattern (move history + undo), Observer (check detection), Factory (piece creation), and correct use of composition (Board contains Squares, Squares reference Pieces). It tests your ability to model a domain you know well in an object-oriented way without over-engineering it.

The Core Design Challenge

Where do you put move validation logic? Three options: in each Piece subclass (ValidMove()), in the Board class (ValidateMove(piece, position)), or in a separate MoveValidator. Think about this before coding. The answer: each Piece knows its own movement rules (it's part of what makes it that piece). The Board validates that the move doesn't put the moving player's King in check — that's Board-level concern. MoveValidator handles the interaction between both. This is SRP in action.

Key Design Decisions to Articulate in Interview
  • "I'm making Piece abstract with abstract GetValidMoves(Board) — each piece implements its own movement logic"
  • "Command pattern for Move — because Chess requires undo/redo, and every move needs to be reversible"
  • "Observer for check detection — when a King's position changes, all pieces notify a CheckDetector"
  • "Board is not an array of pieces — it's an array of Squares. Square can be empty or contain a Piece. This makes null checks cleaner"
Your Practice Task

Build Chess in 3 sessions: Session 1 — entities and relationships only (no methods, just class skeletons). Session 2 — implement piece movement for King, Queen, Rook (the three different movement types cover all cases). Session 3 — implement the Command pattern for moves with undo. Don't implement the full game — demonstrate the design pattern knowledge clearly.

After completing each problem, do a "post-mortem" in your design journal: What design decisions did I make? Were there alternatives I didn't consider? What would change if the requirements doubled in complexity? What pattern did I use that felt forced? This reflection is what converts practice into genuine expertise.

The sequence for practice problems: Parking Lot → Library System → ATM/Vending Machine → Elevator → BookMyShow → Chess → Food Delivery → Splitwise. Do them in this order. Each one introduces one new complexity layer.

Phase 4 exit criteria: Can you design any L1/L2 problem fully (entities, relationships, patterns, key code) in 45 minutes without looking at notes? Can you name at least 2 design alternatives for each major decision and justify why you chose the one you did?

Interview Communication Patterns
①
Open Every Problem the Same Way
Say: "Before I start designing, let me ask a few clarifying questions." Then ask your 3–5 questions. Then say: "Based on that, let me identify the key entities first." This scripted opening buys you thinking time, signals a structured approach, and immediately impresses the interviewer. Practise this opening until it comes out automatically under pressure.
②
Name Every Decision Out Loud
Don't just write code. Say: "I'm making Vehicle abstract here because Car, Truck, and Motorcycle are all vehicles but have different weight and size constraints that will affect which spots they can use." Interviewers can't see your reasoning — you have to speak it. Every class you create, every interface you define, every pattern you apply: name it and justify it. This is the single highest-impact thing you can do in an LLD interview.
③
Handle Pushback Gracefully
When the interviewer says "why didn't you use X instead?" — never immediately capitulate and change your design. Say: "That's a good point. The tradeoff between my approach and X is: mine optimises for Y, yours would be better if Z. Given the requirements we discussed, I went with mine — but I'm open to reconsidering if Z is a priority." This response shows confidence, system thinking, and collaboration. All are senior engineer signals.
④
Know When to Write Code vs Keep Designing
In a 45-minute interview, spend the first 15 minutes on Steps 1–4 (requirements, entities, relationships, patterns). Then spend 25 minutes on code — but only implement the 3–4 classes that best demonstrate your design thinking. Save 5 minutes at the end for "if I had more time, I'd also add X, Y, Z". Never let code writing consume the full session — the design discussion is what scores the most points.
Common Interview Questions — With Frameworks
"Explain the difference between abstract class and interface in C#. When do you use each?" ▼

Framework: Interface = pure contract, no implementation, multiple inheritance. Abstract class = partial implementation, single inheritance, can have state and constructors.

Use interface when: you're defining a capability that unrelated classes can have (IDisposable, IComparable). Use abstract class when: you want to share implementation across related types and there's a genuine "is-a" hierarchy.

Gold answer addition: "In .NET, I lean towards interfaces because they're more testable and support DI better. I use abstract classes when I have genuine shared behaviour that all subclasses need — like a base BaseRepository that handles connection management, letting concrete repos focus only on their queries."

"Design a Notification System. Walk me through your approach." ▼

Framework: Step 1: Ask — what channels (Email/SMS/Push)? Multiple recipients? Priority levels? Templating? Retry on failure? Step 2: Entities — Notification, NotificationRequest, INotificationChannel, NotificationService, NotificationTemplate. Step 3: Relationships — NotificationService uses INotificationChannel (association, DIP). NotificationRequest aggregates channels and recipients. Step 4: Patterns — Strategy (channel-per-implementation), Observer (for tracking delivery status), Builder (building complex NotificationRequest). Step 5: Show INotificationChannel interface first, then Email/SMS implementations, then NotificationService injecting all channels via DI.

"What's the difference between Strategy and State patterns?" ▼

The answer most engineers get wrong: "Strategy is about choosing an algorithm, State is about object state." That's incomplete.

The complete answer: In Strategy, the strategies are independent of each other — they don't know about each other. The context holds the strategy and can switch between them. In State, each state object can trigger transitions to other states — they know about each other's existence (or at least about the transitions). The state machine logic lives in the state objects, not just in the context.

Example: Payment Strategy — each payment method doesn't know about other methods. Order State — ProcessingState knows it can transition to ShippedState or CancelledState. The state itself drives the transitions.

"How would you handle concurrency in a Parking Lot — two cars arriving at the same spot simultaneously?" ▼

This tests whether you think about thread safety in LLD. The answer has two layers:

Design layer: Spot assignment should be an atomic operation. Use a locking mechanism at the Floor level (not the Lot level, to allow parallel access to different floors). ParkingSpot's status transition should be atomic — check-then-act as one operation.

C# layer: Use lock on spot assignment, or use Interlocked operations on a status integer, or use a concurrent data structure. ConcurrentDictionary for available spots is a good choice. Say: "In a distributed system, I'd use a distributed lock (Redis) or an optimistic concurrency approach with a version field."

Mentioning this automatically — even before being asked — is a senior engineer signal.

Learn From Others' Failures
The 10 Mistakes That Kill LLD Prep
I've seen smart engineers fail LLD interviews not because they lacked knowledge, but because of specific, fixable habits. Here are the most common ones — and how to avoid them.

Read these once. Then come back and read them again after Week 8. You'll recognise yourself in at least 3 of them. That recognition is valuable — most engineers don't become aware of these patterns until an interviewer's feedback reveals them.

01
Memorising patterns, not recognising them
You can recite "Singleton ensures one instance" but freeze when asked "how would you ensure only one database connection pool?" Same concept, different framing. Fix: for each pattern, write 5 different real-world scenarios where you'd apply it.
02
Coding without designing
Opening VS Code before thinking about entities and relationships. The result: messy code that doesn't reflect a coherent design. Fix: enforce a rule — no IDE until paper design is complete. Every time, even for small problems.
03
Over-engineering with too many patterns
Using 6 patterns in a Parking Lot to "show knowledge". Interviewers see this and think "over-engineered, would add complexity to a real codebase." Fix: use patterns only where they solve a real problem you can articulate. One forced pattern tanks your score more than zero patterns.
04
Not asking clarifying questions
"I'll just design the general case." Engineers who do this either miss key requirements or design something too complex. Asking questions signals maturity. Skipping them signals overconfidence or unpreparedness. Fix: always ask at least 3 questions before touching the design.
05
Designing silently
Staring at the whiteboard/screen and thinking without speaking. The interviewer can't see your reasoning — they only see the output. Fix: narrate everything. "I'm thinking about whether Vehicle should be abstract or an interface. I'm going with abstract because I want to share the plate number property across all types..."
06
Ignoring edge cases
Happy-path-only design. What happens when a user tries to withdraw more than their balance? When a spot is double-booked? When an elevator request comes from a non-existent floor? Fix: after your initial design, explicitly ask "what are 3 edge cases this design handles or doesn't handle?"
07
Passive reading without implementation
Reading about patterns and thinking "I get it" without writing code. You don't get it until your fingers produce it without help. Fix: close every resource after reading a concept. Open IDE. Implement from memory. If you can't — you haven't learned it yet.
08
Never doing mock interviews until late
"I'll start mocks when I'm ready." You're never ready until you've done mocks. The performance layer — talking while designing under time pressure — is a completely separate skill you can only build by doing it. Fix: start mocks at Week 11, not Week 15.
09
Rebuilding the same 3 problems forever
Doing Parking Lot 10 times because it feels safe. You're optimising for comfort, not growth. Fix: once you can do a problem in 45 minutes clean, move to a harder one. New problems force genuine generalisation of design thinking. Return to old ones for speed drills only.
10
Not reviewing what went wrong
Finishing a practice session and moving on without reflection. You'll make the same mistakes next time. Fix: after every practice session — 5 minutes of journaling. What design choice was wrong? What would I do differently? What question should I have asked at the start?
